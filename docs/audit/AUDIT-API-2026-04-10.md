## CRÍTICO

### [C-04] 6 Endpoints de Autenticação e Conta em Falta no Backend
**Ficheiro Frontend:** `src/frontend/src/features/identity-access/api/identity.ts` (linhas 141–169)  
**Módulo Backend:** `identityaccess` — endpoints não implementados  

**Problema:**  
O frontend tem funções API preparadas para os seguintes fluxos que **não existem no backend**:

| Função Frontend | Rota Esperada | Método | Status Backend |
|----------------|---------------|--------|----------------|
| `activateAccount()` | `/identity/auth/activate` | POST | ❌ Não existe |
| `forgotPassword()` | `/identity/auth/forgot-password` | POST | ❌ Não existe |
| `resetPassword()` | `/identity/auth/reset-password` | POST | ❌ Não existe |
| `resendMfaCode()` | `/identity/auth/mfa/resend` | POST | ❌ Não existe |
| `getInvitationDetails()` | `/identity/invitations/{token}` | GET | ❌ Não existe |
| `acceptInvitation()` | `/identity/invitations/accept` | POST | ❌ Não existe |

**Impacto:**
- Activação de conta de novos utilizadores: quebrado
- Recuperação de password: quebrada
- Reset de password: quebrado
- Reenvio de código MFA: quebrado
- Sistema de convites: completamente não funcional

**Correcção — Backend:**

Criar os seguintes handlers e endpoints no módulo `identityaccess`:

```
Application/Features/
├── ActivateAccount/
│   ├── ActivateAccount.cs (Command + Handler)
│   └── ActivateAccountValidator.cs
├── ForgotPassword/
│   ├── ForgotPassword.cs
│   └── ForgotPasswordValidator.cs
├── ResetPassword/
│   ├── ResetPassword.cs
│   └── ResetPasswordValidator.cs
├── ResendMfaCode/
│   └── ResendMfaCode.cs
└── Invitations/
    ├── GetInvitationDetails/
    │   └── GetInvitationDetails.cs
    └── AcceptInvitation/
        ├── AcceptInvitation.cs
        └── AcceptInvitationValidator.cs
```

Mapear em `AuthEndpoints.cs`:
```csharp
authGroup.MapPost("/activate", ...)
    .AllowAnonymous();

authGroup.MapPost("/forgot-password", ...)
    .AllowAnonymous()
    .RequireRateLimiting("auth-sensitive");

authGroup.MapPost("/reset-password", ...)
    .AllowAnonymous()
    .RequireRateLimiting("auth-sensitive");

authGroup.MapPost("/mfa/resend", ...)
    .AllowAnonymous()
    .RequireRateLimiting("auth-sensitive");
```

Criar `InvitationEndpoints.cs`:
```csharp
var invGroup = app.MapGroup("/identity/invitations");
invGroup.MapGet("/{token}", ...)
    .AllowAnonymous();
invGroup.MapPost("/accept", ...)
    .AllowAnonymous();
```

Requisitos de segurança:
- Tokens de reset/activate devem expirar em 24h
- Tokens de convite devem expirar conforme configuração do tenant
- Todas as operações devem emitir `SecurityEvent`
- Rate limiting obrigatório nos endpoints de reset/forgot

## ALTO

### [A-06] DTO Mismatch — Endpoint de Correlação de Incidentes
**Ficheiro Frontend:** `src/frontend/src/features/operations/api/incidents.ts` (linha 545)  
**Ficheiro Backend:** `src/modules/operationalintelligence/.../IncidentEndpointModule.cs` (linha 204)  

**Problema:**  
O frontend passa `incidentId` como string, mas o backend define a rota como `/{incidentId:guid}` — o ASP.NET Core rejeita strings que não sejam GUIDs válidos com 404.

```typescript
// Frontend — passa string:
const response = await client.post(`/incidents/${incidentId}/correlate`);
```

```csharp
// Backend — espera GUID:
group.MapPost("/{incidentId:guid}/correlate", ...);
```

**Correcção (opção recomendada — manter GUID no backend):**

No frontend, garantir que `incidentId` é sempre tratado como GUID:
```typescript
// Validar antes de chamar
if (!isValidUUID(incidentId)) {
  throw new Error('Invalid incident ID format');
}
```

Alternativa (se IDs de incidente podem não ser GUIDs):
```csharp
// Backend — remover constraint de tipo:
group.MapPost("/{incidentId}/correlate", ...);
// Validar no handler com FluentValidation
```

## MÉDIO

### [M-09] Routing Order — `/runbooks/suggest` vs `/{runbookId}`
**Ficheiro Backend:** `src/modules/operationalintelligence/.../RunbookEndpointModule.cs` (linha 95)  

**Problema:**  
ASP.NET Core resolve rotas em ordem de definição. Se `/{runbookId}` estiver mapeado antes de `/suggest`, requisições para `/runbooks/suggest` serão capturadas pela rota paramétrica com `runbookId = "suggest"`, resultando em 404 ou erro.

**Verificação:**  
Confirmar que no `RunbookEndpointModule.cs`, `/suggest` está mapeado ANTES de `/{runbookId}`:

```csharp
// CORRECTO — suggest ANTES da rota paramétrica:
group.MapGet("/suggest", ...);        // linha X
group.MapGet("/{runbookId:guid}", ...); // linha X+N

// ERRADO — suggest DEPOIS:
group.MapGet("/{runbookId}", ...);    // captura "suggest"
group.MapGet("/suggest", ...);        // nunca é atingido
```

**Correcção:** Reordenar as rotas e adicionar `:guid` constraint ao `/{runbookId}` para evitar que capture strings não-GUID.
