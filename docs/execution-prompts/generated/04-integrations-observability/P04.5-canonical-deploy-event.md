# PROMPT — Define canonical DeployEvent domain entity with multi-source mappers

## 2. Objective
Define a canonical `CanonicalDeployEvent` domain entity in the OperationalIntelligence module that normalizes deploy data from multiple ingestion sources (GitHub Actions, generic webhook, manual entry) into a single consistent model. Implement mapper interfaces and concrete mappers so that downstream correlation (incident↔deploy, change tracking, release linking) operates on a uniform data structure regardless of the deploy event's origin.

## 3. Problem actual
Each CI/CD connector and ingestion source produces different payload formats:
- GitHub Actions sends `workflow_run` with `head_sha`, `head_branch`, `conclusion`
- Generic webhooks send arbitrary JSON with varying field names (`commit`, `version`, `env`, `service_name`)
- Future connectors (Jenkins, GitLab, Azure DevOps) will have their own formats
- Manual deploy entries may come from the UI with NexTraceOne-native field names

Without a canonical model, downstream consumers must understand every source format individually. This makes incident↔deploy correlation (P01.1), change tracking, blast radius computation, and release correlation (P04.6) fragile and unmaintainable.

Currently, `CanonicalDeployEvent` does not exist in code — it appears only in planning documentation (`docs/execution-prompts/EXECUTION-PROMPTS-SEQUENCE.md`, `docs/execution-prompts/PROBLEM-TO-PROMPT-MAPPING.csv`).

The OperationalIntelligence module has existing domain structure:
- `src/modules/operationalintelligence/NexTraceOne.OperationalIntelligence.Domain/Runtime/Entities/` — runtime entities
- `src/modules/operationalintelligence/NexTraceOne.OperationalIntelligence.Domain/Runtime/Enums/` — runtime enums
- `src/modules/operationalintelligence/NexTraceOne.OperationalIntelligence.Domain/Runtime/Events/` — domain events
- `src/modules/operationalintelligence/NexTraceOne.OperationalIntelligence.Domain/Runtime/Ports/` — port interfaces
- `src/modules/operationalintelligence/NexTraceOne.OperationalIntelligence.Application/` — application handlers

## 8. Passos obrigatórios
1. **Create `CanonicalDeployEvent` domain entity:**
   - `Id` (Guid) — strongly typed ID preferred if pattern exists in codebase
   - `ServiceId` (Guid?) — nullable, linked to Catalog service if matched
   - `ServiceName` (string) — resolved or extracted service name
   - `Environment` (string) — target deployment environment
   - `Version` (string?) — application version or tag
   - `CommitSha` (string?) — source commit
   - `Branch` (string?) — source branch
   - `Source` (DeployEventSource enum)
   - `SourceExternalId` (string?) — external system's ID for this deploy (e.g., GitHub run ID)
   - `Status` (DeployEventStatus enum)
   - `DeployedAt` (DateTimeOffset) — when the deploy occurred
   - `IngestedAt` (DateTimeOffset) — when NexTraceOne received the event
   - `IngestionExecutionId` (Guid?) — link back to source IngestionExecution
   - `Metadata` (string?) — JSON blob for source-specific extra data
   - `TenantId` (Guid) — tenant isolation
   - Standard audit fields: `CreatedAt`, `CreatedBy`
2. **Create `DeployEventSource` enum:**
   - `GitHubActions`, `Jenkins`, `GitLab`, `AzureDevOps`, `Manual`, `Generic`
3. **Create `DeployEventStatus` enum:**
   - `Success`, `Failure`, `Cancelled`, `InProgress`, `Unknown`
4. **Create `IDeployEventMapper` interface:**
   ```csharp
   public interface IDeployEventMapper
   {
       DeployEventSource SupportedSource { get; }
       CanonicalDeployEvent Map(IngestionExecutionDto execution);
   }
   ```
   - `IngestionExecutionDto` is a DTO from Integrations contracts (or a local DTO if contracts not available)
5. **Implement `GitHubActionsDeployEventMapper`:**
   - Map from GitHub Actions parsed ingestion data (workflow name as ServiceName, head_branch as Branch, head_sha as CommitSha, conclusion as Status, etc.)
   - Set `Source = DeployEventSource.GitHubActions`
   - Set `SourceExternalId` to the GitHub workflow run ID
6. **Implement `GenericDeployEventMapper`:**
   - Map from generic parsed ingestion data (P04.3 parsed fields)
   - Set `Source = DeployEventSource.Generic`
   - Handle missing fields gracefully (nullable properties)
7. **Create `ICanonicalDeployEventRepository` port:**
   - `AddAsync(CanonicalDeployEvent, CancellationToken)`
   - `GetByIdAsync(Guid, CancellationToken)`
   - `GetByServiceIdAsync(Guid serviceId, DateTimeOffset from, DateTimeOffset to, CancellationToken)`
   - `GetByCommitShaAsync(string commitSha, CancellationToken)`
8. **Implement EF repository:**
   - Use IncidentDbContext or create a dedicated DbSet in the OperationalIntelligence context
   - Add EF configuration for the entity
9. **Create `ProcessDeployEvent` command handler:**
   - Accept: `IngestionExecutionId` (Guid), `Source` (DeployEventSource)
   - Load ingestion execution data (via cross-module query or Integrations contracts)
   - Select appropriate mapper based on `Source`
   - Map to `CanonicalDeployEvent`
   - Persist via repository
   - Publish `DeployEventProcessed` domain event
   - Return `Result<Guid>` with the deploy event ID
10. **Add unit tests (≥7):**
    - `GitHubActionsDeployEventMapper` maps all fields correctly from valid data
    - `GitHubActionsDeployEventMapper` handles missing optional fields
    - `GenericDeployEventMapper` maps all fields correctly from parsed ingestion
    - `GenericDeployEventMapper` handles minimal data (only service name)
    - `ProcessDeployEvent` handler selects correct mapper
    - `ProcessDeployEvent` handler persists and publishes event
    - `ProcessDeployEvent` handler returns failure if ingestion execution not found

## 13. Riscos e cuidados
- **MEDIUM RISK:** Cross-module data access — `ProcessDeployEvent` needs to read `IngestionExecution` from the Integrations module. Use the Integrations contracts project (`NexTraceOne.Integrations.Contracts`) if it exposes a query interface. If not, create a local DTO and a query that reads via a shared service or direct cross-context query with clear documentation of the coupling.
- **MEDIUM RISK:** Adding `CanonicalDeployEvent` to IncidentDbContext (or its own context) without migration will cause EF model mismatch warnings. Mark as `[NotMapped]` initially or add the DbSet knowing a migration will follow.
- **CAUTION:** `ServiceId` matching — the mapper should NOT fail if no matching service is found. Leave `ServiceId` null and set `ServiceName` from the ingestion data. Service matching can be refined later.
- **CAUTION:** `Metadata` JSON field — validate JSON before storing. Do not store raw untrusted input without sanitization.
- **CAUTION:** Mapper selection — use a factory or DI-based strategy pattern to resolve the correct mapper. Do not use switch statements that require modification when new sources are added.
