# PROMPT — Validate OTEL → Collector → ClickHouse pipeline end-to-end

## 1. Identification
- **Prompt ID:** P04.2
- **Phase:** 04-integrations-observability
- **Execution order:** 34
- **Type:** validation
- **Area:** infrastructure / observability
- **Module:** platform
- **Bounded context:** otel-pipeline
- **Can run in parallel:** no
- **Risk level:** medium

## 2. Objective
Validate end-to-end that the OpenTelemetry pipeline works: application emits traces/metrics → OTEL Collector receives them → Collector exports to ClickHouse → data is queryable in ClickHouse. Identify and fix any broken links in this pipeline. Document the validated state.

## 3. Problem actual
Per forensic audit: "ClickHouse schema defined but pipeline integration incomplete. OTEL endpoint not validated end-to-end." The project has:

- An OTEL Collector configuration at `build/otel-collector/otel-collector.yaml`
- A Docker Compose telemetry stack at `build/otel-collector/docker-compose.telemetry.yaml`
- ClickHouse documentation at `docs/observability/providers/clickhouse.md` and `docs/architecture/clickhouse-baseline-strategy.md`
- Application-level OpenTelemetry instrumentation in `src/building-blocks/NexTraceOne.BuildingBlocks.Observability/`
- Observability readiness tests at `tests/building-blocks/NexTraceOne.BuildingBlocks.Observability.Tests/Telemetry/OpenTelemetryReadinessTests.cs`

However, no one has confirmed the full pipeline works: traces generated by the application flow through the collector and land in ClickHouse tables. The collector config may not be correctly routing to ClickHouse. The ClickHouse schema may not match the collector's expected output format.

## 4. Resultado esperado
- Docker Compose stack starts successfully with: NexTraceOne API, OTEL Collector, ClickHouse, PostgreSQL
- Application generates traces when API endpoints are called
- OTEL Collector receives traces (confirmed via collector logs or health endpoint)
- ClickHouse contains trace data (confirmed via SQL query)
- Any broken configuration is fixed
- A validation report documents what works, what was fixed, and any remaining gaps

## 5. Escopo permitido
- **Files:**
  - `docker-compose.yml` and `docker-compose.override.yml` — to ensure services are correctly wired
  - `build/otel-collector/otel-collector.yaml` — to fix collector pipeline configuration
  - `build/otel-collector/docker-compose.telemetry.yaml` — to fix telemetry stack if needed
  - ClickHouse init scripts (if they exist under `build/` or `infra/`)
  - Documentation files under `docs/observability/`
- **Actions:** Start services, make API calls, query ClickHouse, fix collector config, write validation report

## 6. Escopo proibido

- Do NOT install new NuGet packages in the application
- Do NOT modify PostgreSQL schemas or migrations
- Do NOT change the application's trace/metric generation logic
- Do NOT add Grafana, Prometheus, or any visualization layer — this is about data pipeline, not dashboards

## 7. Ficheiros principais candidatos a alteração

## 8. Passos obrigatórios
1. **Inventory the current stack:**
   - Check `docker-compose.yml` for otel-collector and clickhouse service definitions
   - Check `build/otel-collector/docker-compose.telemetry.yaml` for telemetry stack
   - Check `build/otel-collector/otel-collector.yaml` for collector pipeline (receivers, processors, exporters)
   - Search for ClickHouse init scripts (`.sql` files with `CREATE TABLE` for traces/metrics)
2. **Verify collector configuration:**
   - Confirm the collector has an OTLP receiver on port 4317 (gRPC) and/or 4318 (HTTP)
   - Confirm the collector has a ClickHouse exporter configured with correct DSN, database name, and table names
   - If the ClickHouse exporter is missing or misconfigured, fix it using the `clickhouseexporter` config
3. **Verify ClickHouse schema:**
   - Confirm ClickHouse has tables matching what the collector exporter expects (typically `otel_traces`, `otel_logs`, `otel_metrics_*`)
   - If schemas are missing, create init SQL or use the collector's auto-create-tables feature
4. **Start the full stack:**
   - Run `docker compose up` (or `docker compose -f docker-compose.yml -f build/otel-collector/docker-compose.telemetry.yaml up`)
   - Confirm all services start and are healthy
5. **Generate test traces:**
   - Make several API calls to the NexTraceOne API (e.g., `GET /api/v1/catalog/services`, health check endpoints)
   - Wait 10-30 seconds for traces to propagate
6. **Verify collector is receiving data:**
   - Check collector logs for received spans/metrics
   - If collector has a health or metrics endpoint, query it
7. **Verify ClickHouse has data:**
   - Connect to ClickHouse (via `docker exec` or ClickHouse client)
   - Run: `SELECT count(*) FROM otel_traces` (or appropriate table name)
   - Run: `SELECT * FROM otel_traces LIMIT 10` to inspect data shape
8. **Document results:**
   - If everything works: document the working configuration
   - If something was broken: document what was broken and how it was fixed
   - If something remains broken: document the gap and proposed fix
9. **Create validation report** at `docs/observability/pipeline-validation-report.md`

## 13. Riscos e cuidados
- **MEDIUM RISK:** The OTEL Collector ClickHouse exporter (`clickhouseexporter`) may require a specific collector distribution (e.g., `otel/opentelemetry-collector-contrib`). Verify the Docker image used for the collector includes the ClickHouse exporter.
- **MEDIUM RISK:** ClickHouse table schema must match the exporter's expected format. If using the contrib exporter, it may auto-create tables — verify this behavior and whether it conflicts with any pre-existing init scripts.
- **CAUTION:** Docker Compose networking — the collector must be able to reach ClickHouse by service name (e.g., `clickhouse:9000`). Verify they are on the same Docker network.
- **CAUTION:** ClickHouse default port is 9000 (native protocol) or 8123 (HTTP). The collector exporter config must use the correct port and protocol.
- **CAUTION:** Large trace volumes during testing could fill ClickHouse storage quickly — use `LIMIT` on test data and avoid stress testing in this prompt.
