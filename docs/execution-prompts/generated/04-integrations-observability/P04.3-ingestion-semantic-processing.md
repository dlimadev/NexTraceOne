# PROMPT — Implement semantic processing of ingestion payloads

## 1. Identification
- **Prompt ID:** P04.3
- **Phase:** 04-integrations-observability
- **Execution order:** 35
- **Type:** feature-implement
- **Area:** backend
- **Module:** integrations
- **Bounded context:** ingestion
- **Can run in parallel:** yes
- **Risk level:** medium

## 2. Objective
Make the Ingestion API process deploy/change payloads semantically instead of just recording metadata. Currently, payloads are accepted but only metadata is persisted with `processingStatus: "metadata_recorded"`. This prompt implements actual payload parsing, field extraction, and enriched persistence so that downstream consumers (Change Intelligence, Operational Intelligence) can use real data.

## 3. Problem actual
Per forensic audit: "Ingestion API: 5 endpoints exist with processingStatus: 'metadata_recorded' — payload not processed. Payloads accepted but only metadata persisted."

The ingestion endpoints in `src/platform/NexTraceOne.Ingestion.Api/Program.cs` accept deploy, change, incident, metric, and log event payloads. However, the processing pipeline is a stub: it creates an `IngestionExecution` record with raw metadata (source, timestamp, status) but does NOT parse the payload JSON to extract semantic fields like service name, environment, version, commit SHA, change type, etc.

Key code locations:
- `src/platform/NexTraceOne.Ingestion.Api/Program.cs` — 5 ingestion endpoints, all set `processingStatus = "metadata_recorded"`
- `src/modules/integrations/NexTraceOne.Integrations.Domain/Entities/IngestionExecution.cs` — domain entity with basic fields
- `src/modules/integrations/NexTraceOne.Integrations.Application/Features/ReprocessExecution/ReprocessExecution.cs` — reprocessing handler (also likely stubbed)
- `src/modules/integrations/NexTraceOne.Integrations.Application/Abstractions/IIngestionExecutionRepository.cs` — repository interface

Without semantic processing, deploy events cannot feed Change Intelligence, incidents cannot be correlated with deploys, and the full traceability chain (P04.5, P04.6) cannot function.

## 4. Resultado esperado
- Ingestion payloads for deploy and change events are parsed to extract: service name, environment, version, timestamp, change type, commit SHA, source system, and any additional metadata
- `IngestionExecution` entity is enriched with parsed fields (or a linked entity stores parsed data)
- `processingStatus` transitions from `"metadata_recorded"` to `"processed"` when parsing succeeds
- `"metadata_recorded"` remains as fallback when payload parsing fails (graceful degradation)
- Domain event is published after successful processing (for downstream consumption)
- Unit tests cover payload parsing for different formats
- Reprocess handler (`ReprocessExecution`) also uses the new parsing logic

## 8. Passos obrigatórios
1. **Inspect the current ingestion flow:**
   - Read `src/platform/NexTraceOne.Ingestion.Api/Program.cs` to understand all 5 ingestion endpoints
   - Read `src/modules/integrations/NexTraceOne.Integrations.Domain/Entities/IngestionExecution.cs` to understand current entity fields
   - Read `src/modules/integrations/NexTraceOne.Integrations.Application/Features/ReprocessExecution/ReprocessExecution.cs` to understand reprocessing
2. **Define the payload parser abstraction:**
   - Create `IIngestionPayloadParser` interface with method: `ParseDeployPayload(string rawPayload) → IngestionParsedResult`
   - `IngestionParsedResult` should be a value object with: `ServiceName`, `Environment`, `Version`, `CommitSha`, `ChangeType`, `Timestamp`, `AdditionalMetadata` (Dictionary<string, string>), `IsSuccessful` (bool), `ErrorMessage` (string?)
3. **Implement `GenericIngestionPayloadParser`:**
   - Parse JSON payload looking for well-known fields (case-insensitive): `service`, `serviceName`, `service_name`, `environment`, `env`, `version`, `commitSha`, `commit_sha`, `commit`, `changeType`, `change_type`, `type`, `timestamp`, `deployedAt`, `deployed_at`
   - Return `IsSuccessful = false` with error message if JSON is malformed or no recognizable fields found
4. **Enrich `IngestionExecution` entity:**
   - Add nullable string properties: `ParsedServiceName`, `ParsedEnvironment`, `ParsedVersion`, `ParsedCommitSha`, `ParsedChangeType`
   - Add `ParsedAt` (DateTimeOffset?) to record when parsing occurred
   - Add method: `MarkAsProcessed(IngestionParsedResult result)` that sets parsed fields and updates `ProcessingStatus`
   - Add method: `MarkAsFailed(string errorMessage)` for parsing failure
5. **Create `ProcessIngestionPayload` command handler:**
   - Receives `IngestionExecutionId`
   - Loads the execution from repository
   - Invokes `IIngestionPayloadParser.ParseDeployPayload` with the raw payload
   - Calls `MarkAsProcessed()` or `MarkAsFailed()` on the entity
   - Persists changes
   - Publishes a domain event: `IngestionPayloadProcessed` (with execution ID, parsed service, environment, version)
6. **Update ingestion endpoints in `Program.cs`:**
   - After the existing metadata recording step, dispatch the `ProcessIngestionPayload` command
   - For deploy and change events: invoke processing immediately
   - For other event types (incident, metric, log): keep `metadata_recorded` for now (parsing scope is deploy/change only)
7. **Update `ReprocessExecution` handler:**
   - Instead of just resetting status, invoke the `ProcessIngestionPayload` flow
8. **Add unit tests:**
   - Valid deploy payload with all fields → `IsSuccessful = true`, all fields parsed
   - Deploy payload with minimal fields (only service and version) → `IsSuccessful = true`, partial parse
   - Malformed JSON → `IsSuccessful = false`, error message set
   - Payload with no recognizable fields → `IsSuccessful = false`
   - `MarkAsProcessed` correctly updates entity state
   - `MarkAsFailed` correctly sets error and keeps metadata_recorded state

## 13. Riscos e cuidados
- **MEDIUM RISK:** Adding properties to `IngestionExecution` entity without a corresponding migration will cause runtime errors if EF Core tries to map them. Use `[NotMapped]` attribute on all new parsed properties initially so they are ignored by EF Core. Document that a follow-up migration prompt must: (1) remove `[NotMapped]`, (2) add EF configuration for the new columns, (3) generate and apply the migration. This approach aligns with Section 6 "Escopo proibido" — no migrations in this prompt.
- **MEDIUM RISK:** The `ProcessIngestionPayload` handler must be resilient to failures — if parsing throws an unhandled exception, the ingestion endpoint should NOT fail. Wrap parsing in try/catch and fall back to `metadata_recorded`.
- **CAUTION:** Field name matching must be case-insensitive and flexible to accommodate different CI/CD systems' payload formats.
- **CAUTION:** The raw payload may be very large — do not log the entire payload. Log only the extracted fields and any errors.
- **CAUTION:** Domain event publishing — if the outbox pattern (P02.1) is not yet implemented, publish directly via MediatR `IPublisher`. Add a TODO comment noting future migration to outbox.
