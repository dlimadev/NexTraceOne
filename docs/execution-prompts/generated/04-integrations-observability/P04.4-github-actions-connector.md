## 2. Objective
Implement the first real CI/CD connector: a GitHub Actions webhook receiver that creates deploy events from `workflow_run.completed` events. This replaces the existing GitHub Actions connector stub with a functional implementation that receives, verifies, parses, and persists deploy data from GitHub Actions workflows.

## 3. Problem actual
Per forensic audit: "CI/CD Connectors: GitLab, Jenkins, GitHub Actions, Azure DevOps all stubs (no implementation). Change Intelligence depends on deploy events but no integration works."

The Integrations module has infrastructure for connectors:
- `src/modules/integrations/NexTraceOne.Integrations.Domain/Entities/` — connector entities exist
- `src/modules/integrations/NexTraceOne.Integrations.Application/Features/ListIntegrationConnectors/` — list handler exists
- `src/modules/integrations/NexTraceOne.Integrations.Application/Features/GetIntegrationConnector/` — get handler exists
- `src/modules/integrations/NexTraceOne.Integrations.Application/Features/RetryConnector/` — retry handler exists
- `src/modules/integrations/NexTraceOne.Integrations.Application/Abstractions/IIntegrationConnectorRepository.cs` — repository interface exists

However, no connector actually processes incoming webhook payloads from any CI/CD system. The Change Intelligence module depends on deploy events to track changes, compute blast radius, and correlate incidents — none of which can function without at least one working connector.

## 4. Resultado esperado
- A new webhook endpoint: `POST /api/v1/integrations/webhooks/github-actions` that receives GitHub Actions webhook payloads
- Webhook signature verification using GitHub's HMAC-SHA256 (`X-Hub-Signature-256` header)
- Parsing of `workflow_run.completed` event payloads to extract: repository name/URL, branch, commit SHA, workflow name, conclusion (success/failure/cancelled), triggered timestamp, actor
- Creation or update of a Connector entity representing the GitHub repository
- Creation of an `IngestionExecution` record with parsed deploy metadata and `processingStatus: "processed"`
- Optional: matching commit SHA or branch to a known service in Catalog (by repository URL)
- Unit tests for payload parsing and signature verification
- Integration test for the webhook endpoint

## 8. Passos obrigatórios
1. **Understand the GitHub Actions webhook payload format:**
   - `workflow_run.completed` event sends: `action`, `workflow_run.id`, `workflow_run.name`, `workflow_run.head_branch`, `workflow_run.head_sha`, `workflow_run.conclusion`, `workflow_run.created_at`, `workflow_run.updated_at`, `workflow_run.html_url`, `repository.full_name`, `repository.html_url`, `sender.login`
   - Reference: GitHub docs for workflow_run webhook event
2. **Create payload DTOs (`GitHubActionsPayload.cs`):**
   - `GitHubActionsWebhookPayload` — top-level: `Action` (string), `WorkflowRun` (object), `Repository` (object), `Sender` (object)
   - `GitHubWorkflowRun` — `Id` (long), `Name` (string), `HeadBranch` (string), `HeadSha` (string), `Conclusion` (string), `CreatedAt` (DateTimeOffset), `UpdatedAt` (DateTimeOffset), `HtmlUrl` (string)
   - `GitHubRepository` — `FullName` (string), `HtmlUrl` (string)
   - `GitHubSender` — `Login` (string)
   - Use `System.Text.Json` naming policy `JsonNamingPolicy.SnakeCaseLower` for deserialization
3. **Implement webhook signature verification (`GitHubWebhookSignatureVerifier.cs`):**
   - Read `X-Hub-Signature-256` header from request
   - Compute HMAC-SHA256 of request body using webhook secret from configuration (`Integrations:GitHub:WebhookSecret`)
   - Compare using constant-time comparison to prevent timing attacks
   - Return `Result<bool>` with failure if signature is missing or invalid
4. **Create `ReceiveGitHubActionsWebhook` command handler:**
   - Accept: raw payload JSON, signature header value
   - Step 1: Verify signature using `IWebhookSignatureVerifier`
   - Step 2: Deserialize payload to `GitHubActionsWebhookPayload`
   - Step 3: Filter — only process `action == "completed"` events; ignore others
   - Step 4: Extract deploy metadata: service name (from repo full_name), branch, commit SHA, conclusion, timestamp, actor
   - Step 5: Create or update `IntegrationConnector` entity for this GitHub repository
   - Step 6: Create `IngestionExecution` record with parsed deploy data and `processingStatus: "processed"`
   - Step 7: Publish domain event `DeployEventIngested` with extracted metadata
   - Step 8: Return success result
5. **Register the webhook endpoint:**
   - `POST /api/v1/integrations/webhooks/github-actions`
   - No authentication (webhooks are verified by signature) — but rate limiting and IP filtering recommended as TODO
   - Read raw request body for signature verification before deserialization
   - Return `200 OK` on success, `401 Unauthorized` on signature failure, `400 Bad Request` on parse failure
6. **Optional: Service matching:**
   - If `ICatalogModule` or a Catalog query interface is available, query for services where `RepositoryUrl` matches the GitHub repo URL
   - If a match is found, set `ServiceId` on the IngestionExecution
   - If no match, leave `ServiceId` null and log a warning
7. **Add unit tests:**
   - Signature verification: valid signature → true, invalid signature → false, missing header → false
   - Payload parsing: valid `workflow_run.completed` → correct fields extracted
   - Payload filtering: `action != "completed"` → skipped gracefully
   - Handler: valid payload creates IngestionExecution with status "processed"
   - Handler: signature failure returns unauthorized result
8. **Add integration test:**
   - POST to `/api/v1/integrations/webhooks/github-actions` with valid payload and signature
   - Verify 200 response
   - Verify IngestionExecution record created (if in-memory DB available)

## 13. Riscos e cuidados
- **MEDIUM RISK:** Webhook endpoint is publicly accessible — signature verification is the primary security control. Ensure the HMAC comparison is constant-time to prevent timing attacks. Use `CryptographicOperations.FixedTimeEquals()` in .NET.
- **MEDIUM RISK:** GitHub webhook payloads can be large (especially for repos with many workflow runs). Limit request body size at the endpoint level (e.g., 1MB max).
- **CAUTION:** The `workflow_run` event may fire for non-deploy workflows (tests, linting, etc.). Consider adding a configuration option to filter by workflow name patterns.
- **CAUTION:** GitHub may retry webhook delivery if it doesn't receive a 2xx response quickly. Ensure the handler responds fast — offload heavy processing to a background job if needed.
- **CAUTION:** The endpoint should NOT require NexTraceOne authentication (external webhook), but should be protected by signature verification. Ensure the auth middleware skips this endpoint.
- **CAUTION:** Ensure webhook secret is stored securely — in `appsettings.json` under `Integrations:GitHub:WebhookSecret` or via environment variable `INTEGRATIONS__GITHUB__WEBHOOKSECRET`.
