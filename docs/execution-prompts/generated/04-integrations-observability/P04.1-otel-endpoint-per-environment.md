# PROMPT — Replace hardcoded OTEL collector endpoint with environment-configurable value

## 1. Identification
- **Prompt ID:** P04.1
- **Phase:** 04-integrations-observability
- **Execution order:** 33
- **Type:** config-fix
- **Area:** backend / infrastructure
- **Module:** platform
- **Bounded context:** otel-config
- **Can run in parallel:** yes
- **Risk level:** low

## 2. Objective
Replace all hardcoded `localhost:4317` OpenTelemetry collector endpoint references with environment-configurable values, supporting both `appsettings` per environment and the standard `OTEL_EXPORTER_OTLP_ENDPOINT` environment variable override. This ensures the OTEL pipeline works correctly across Development, Pre-Production, and Production environments without code changes.

## 3. Problem actual
Per forensic audit: "OpenTelemetry configured to localhost:4317, hardcoded. Not validated end-to-end." The hardcoded endpoint appears in multiple locations:

- `src/platform/NexTraceOne.ApiHost/appsettings.json` — `"Endpoint": "http://localhost:4317"`, `"OtlpGrpcEndpoint": "http://localhost:4317"`, `"OtlpEndpoint": "http://localhost:4317"`
- `src/building-blocks/NexTraceOne.BuildingBlocks.Observability/Telemetry/Configuration/TelemetryStoreOptions.cs` — default property values hardcoded to `http://localhost:4317`
- `build/otel-collector/otel-collector.yaml` — collector config referencing `localhost:4317`

For any non-Development environment, this endpoint must be configurable. Self-hosted enterprise deployments will have different collector addresses depending on network topology, Docker Compose setup, Kubernetes cluster, or IIS deployment.

## 4. Resultado esperado
- All OTEL endpoint references are configurable via `appsettings.{Environment}.json`
- `appsettings.Development.json` retains `localhost:4317` as the default for local dev
- `appsettings.json` (base) contains a placeholder or empty value, not a hardcoded localhost
- `TelemetryStoreOptions.cs` default values are empty/null, requiring explicit configuration
- `OTEL_EXPORTER_OTLP_ENDPOINT` environment variable overrides the appsettings value when present
- `docker-compose.yml` and `docker-compose.override.yml` pass the correct endpoint for containerized operation
- Application boots successfully in Development mode without any change to developer workflow
- The same configuration pattern works for ApiHost, Ingestion.Api, and BackgroundWorkers
