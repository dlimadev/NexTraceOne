# PROMPT — Link deploy events to Change Intelligence releases for full traceability

## 1. Identification
- **Prompt ID:** P04.6
- **Phase:** 04-integrations-observability
- **Execution order:** 38
- **Type:** feature-implement
- **Area:** backend
- **Module:** operationalintelligence
- **Bounded context:** deploy-correlation
- **Can run in parallel:** no
- **Risk level:** medium

## 2. Objective
Link `CanonicalDeployEvent` entities to Change Intelligence releases, completing the incident→deploy→release→change traceability chain. When an incident occurs, operators should be able to trace back: which deploy caused it → which release that deploy belonged to → which specific changes were in that release. This prompt creates the correlation entity, the matching logic, a query handler, and an API endpoint to retrieve the full correlation chain.

## 3. Problem actual
Even after P01.1 (incident↔change correlation) and P04.5 (canonical deploy events), the full traceability chain is incomplete:

- **Incident → Change** works via P01.1's correlation engine (time-window + service matching)
- **Deploy → CanonicalDeployEvent** works via P04.5's mapper pipeline
- **Deploy → Release → Changes** is MISSING — there is no link between a deploy event and the Change Intelligence release it represents

The Change Intelligence module (`src/modules/changegovernance/`) tracks releases with their associated changes. The `IChangeIntelligenceModule` contract interface exists at `src/modules/changegovernance/NexTraceOne.ChangeGovernance.Contracts/ChangeIntelligence/ServiceInterfaces/IChangeIntelligenceModule.cs`. However, no code correlates deploy events (from the Integrations/Observability pipeline) with releases (from Change Intelligence).

This gap means:
- When an incident occurs, operators can find nearby changes but cannot see which deploy brought those changes
- Blast radius analysis cannot leverage deploy event data
- Production change confidence cannot be computed from actual deploy results
- The release calendar cannot show actual deploy outcomes

## 4. Resultado esperado
- `DeployReleaseCorrelation` domain entity exists linking a deploy event to a release
- `ProcessDeployEvent` handler (from P04.5) is extended to automatically correlate with matching releases
- Matching logic: same service + overlapping version or commit SHA between deploy event and release
- `GetDeployCorrelations` query handler returns linked releases and their changes for a given deploy event
- `GET /api/v1/deploy-events/{id}/correlations` endpoint exposes the correlation data
- Incident correlation (P01.1) is enhanced to also consider deploy events when finding correlated changes
- The full traceability chain is functional: Incident → CorrelatedChanges → DeployEvent → Release → Changes
- Unit tests validate correlation logic

## 8. Passos obrigatórios
1. **Create `DeployReleaseCorrelation` domain entity:**
   - `Id` (Guid) — unique correlation identifier
   - `DeployEventId` (Guid) — FK to `CanonicalDeployEvent`
   - `ReleaseId` (Guid) — FK to ChangeGovernance Release (external cross-module reference, not enforced by EF FK to preserve bounded context isolation per DDD module boundaries)
   - `CorrelationType` (enum: VersionMatch, CommitShaMatch, ServiceAndTimeWindow) — how the correlation was established
   - `ConfidenceLevel` (enum: High, Medium, Low) — confidence in the correlation
   - `CorrelatedAt` (DateTimeOffset) — when the correlation was computed
   - `TenantId` (Guid) — tenant isolation
   - Standard audit fields: `CreatedAt`, `CreatedBy`
2. **Create `IDeployReleaseCorrelationRepository` port:**
   - `AddAsync(DeployReleaseCorrelation, CancellationToken)`
   - `GetByDeployEventIdAsync(Guid deployEventId, CancellationToken)`
   - `ExistsByDeployAndReleaseAsync(Guid deployEventId, Guid releaseId, CancellationToken)` — prevent duplicates
3. **Implement EF repository** with proper configuration
4. **Extend `ProcessDeployEvent` handler (from P04.5):**
   - After persisting the `CanonicalDeployEvent`, invoke correlation logic
   - Call `IChangeIntelligenceModule.GetReleasesByServiceAsync(serviceId, cancellationToken)` (or equivalent method) to find releases for the same service
   - For each release, attempt matching:
     - **CommitShaMatch (HIGH confidence):** deploy event's `CommitSha` matches a release's commit SHA → `ConfidenceLevel.High`
     - **VersionMatch (HIGH confidence):** deploy event's `Version` matches a release's version tag → `ConfidenceLevel.High`
     - **ServiceAndTimeWindow (MEDIUM confidence):** same service + release created within configurable time window (default 1 hour) of deploy → `ConfidenceLevel.Medium`
   - Create `DeployReleaseCorrelation` for each match (skip duplicates)
   - Publish `DeployReleaseCorrelated` domain event with deploy ID, release ID, and confidence
5. **Create `GetDeployCorrelations` query handler:**
   - Accept: `DeployEventId` (Guid)
   - Load all `DeployReleaseCorrelation` records for the deploy event
   - For each correlation, load release details from `IChangeIntelligenceModule` (release name, version, change count, created date)
   - Return response DTO: `DeployCorrelationsResponse` with deploy event summary + list of correlated releases with their metadata
6. **Register API endpoint:**
   - `GET /api/v1/deploy-events/{id}/correlations`
   - Permission: `operationalintelligence:read` (or appropriate permission from existing auth setup)
   - Returns: `DeployCorrelationsResponse` with 200 OK
   - Returns: 404 if deploy event not found
   - Returns: empty correlations array if no matches found (200 with empty list, not 404)
7. **Enhance incident correlation (P01.1 integration):**
   - In the existing `CorrelateIncidentWithChanges` handler (if implemented), add an additional correlation path:
     - For each incident, find deploy events for the same service within the incident's time window
     - For each matching deploy event, look up its release correlations
     - Add changes from those releases to the incident's correlated changes (with `CorrelationType = ViaDeployEvent`)
   - If P01.1 handler does not yet exist, create a TODO comment documenting this future integration point
8. **Add unit tests (≥7):**
   - CommitSha match produces High confidence correlation
   - Version match produces High confidence correlation
   - ServiceAndTimeWindow match produces Medium confidence
   - No matching release → no correlation created (empty result)
   - Duplicate correlation prevention (same deploy + release not correlated twice)
   - `GetDeployCorrelations` returns correct data structure
   - `GetDeployCorrelations` returns empty list for deploy event with no correlations

## 13. Riscos e cuidados
- **MEDIUM RISK:** `IChangeIntelligenceModule` may not expose a method to query releases by service and version/commit. Inspect the interface before implementing. If the required method doesn't exist, create a minimal extension in the contracts project or document the gap.
- **MEDIUM RISK:** Cross-module query performance — if the release query returns many results, correlation matching could be slow. Add time-window limits to the release query (e.g., only releases from the last 30 days).
- **CAUTION:** Tenant isolation — correlation queries must filter by tenant. Ensure `IChangeIntelligenceModule` accepts tenant context or the calling handler provides it.
- **CAUTION:** The `ProcessDeployEvent` handler is growing in responsibility (map + persist + correlate). Consider extracting correlation logic into a separate `CorrelateDeployWithReleases` handler triggered by the `DeployEventProcessed` event — this follows the event-driven pattern and keeps handlers focused.
- **CAUTION:** Version matching may be fuzzy (e.g., `v1.2.3` vs `1.2.3`). Normalize versions before comparison (strip leading `v` or `V`).
- **CAUTION:** CommitSha matching should use prefix matching (at least 7 characters) to handle short SHA references.
