# PROMPT — Add per-module status endpoint checks to staging smoke tests

## 1. Identification
- **Prompt ID:** P05.4
- **Phase:** 05-quality-hardening
- **Execution order:** 42
- **Type:** pipeline-config
- **Area:** ci-cd
- **Module:** platform
- **Bounded context:** smoke-checks
- **Can run in parallel:** yes (group-E)
- **Risk level:** low

## 2. Objective
Enhance the staging deployment smoke test to check each module's status/health endpoint individually, ensuring all modules are operational after deployment instead of just checking the root endpoint.

## 3. Problem actual
Per forensic audit (`tests-quality-pipelines-report.md`): "staging.yml: BOM — migrations, smoke checks" but current smoke checks likely only verify the root endpoint responds. Each NexTraceOne module exposes a status endpoint (e.g., `/api/v1/catalog/status`, `/api/v1/identity/status`). A module could fail silently without being caught.

## 4. Resultado esperado
- Staging smoke test checks ≥10 module status endpoints
- Each check verifies HTTP 200 response with module health info
- Deployment fails if any module status endpoint is unhealthy
- Status check list is documented and maintainable

## 5. Escopo permitido
- **Directories:** `.github/workflows/`, scripts/
- **Files:** `staging.yml`, smoke check script
- **Artifact types:** Workflow YAML, shell script

## 6. Escopo proibido
- Do NOT modify module status endpoints
- Do NOT modify application code
- Do NOT modify production.yml workflow
- Do NOT modify ci.yml or e2e.yml

## 7. Ficheiros principais candidatos a alteração
1. `.github/workflows/staging.yml` — enhance smoke check step
2. `scripts/smoke-check.sh` — create or update dedicated smoke check script (if one doesn't exist)
3. `docs/architecture/` — document module status endpoints list

## 8. Passos obrigatórios
1. Identify all module status endpoints by searching for `status` or `health` endpoint registrations in `*EndpointModule.cs` files
2. Create a list of expected endpoints:
   - `GET /api/v1/catalog/status`
   - `GET /api/v1/identity/status`
   - `GET /api/v1/changes/status`
   - `GET /api/v1/audit/status`
   - `GET /api/v1/governance/status`
   - `GET /api/v1/incidents/status`
   - `GET /api/v1/ai/status`
   - `GET /api/v1/integrations/status`
   - `GET /api/v1/knowledge/status`
   - `GET /api/v1/notifications/status`
3. Create or update `scripts/smoke-check.sh`:
   ```bash
   BASE_URL=${1:-http://localhost:5000}
   FAILED=0
   for endpoint in catalog identity changes audit governance incidents ai integrations knowledge notifications; do
     STATUS=$(curl -s -o /dev/null -w "%{http_code}" --max-time 5 "$BASE_URL/api/v1/$endpoint/status")
     if [ "$STATUS" != "200" ]; then echo "FAIL: $endpoint (HTTP $STATUS)"; FAILED=1; fi
   done
   exit $FAILED
   ```
4. In staging.yml: replace or augment smoke check step with `bash scripts/smoke-check.sh $BASE_URL`
5. Add `timeout-minutes: 5` to the smoke check step
6. Test script locally or document how to test
