# PROMPT — Reduce CS8632 nullable warnings in core modules

## 1. Identification
- **Prompt ID:** P05.3
- **Phase:** 05-quality-hardening
- **Execution order:** 41
- **Type:** code-quality
- **Area:** backend
- **Module:** cross-cutting
- **Bounded context:** nullable
- **Can run in parallel:** yes (group-E)
- **Risk level:** low

## 2. Objective
Reduce the 516 CS8632 nullable reference type warnings in the four highest-maturity modules (Catalog, ChangeGovernance, IdentityAccess, AuditCompliance) to fewer than 100 by adding proper nullable annotations.

## 3. Problem actual
Per forensic audit (`tests-quality-pipelines-report.md`): "516 CS8632 nullable warnings across the solution. Most in high-maturity modules that should be clean." These warnings indicate missing nullable reference type annotations, reducing code clarity and hiding potential null reference bugs in production-ready modules.

## 4. Resultado esperado
- CS8632 warnings in Catalog, ChangeGovernance, IdentityAccess, and AuditCompliance modules reduced from ~400+ to < 100
- Properties that can legitimately be null have `?` annotations
- Properties that should never be null have guard clauses or `= null!` suppressions where appropriate
- DTOs match API contract nullability expectations
- No changes to method signatures in public APIs

## 5. Escopo permitido
- **Modules:** Catalog, ChangeGovernance, IdentityAccess, AuditCompliance only
- **Directories:**
  - `src/modules/catalog/`
  - `src/modules/changegovernance/`
  - `src/modules/identityaccess/`
  - `src/modules/auditcompliance/`
- **Types:** Domain entities, DTOs, handlers, repositories, services

## 6. Escopo proibido
- Do NOT modify Governance module (100% mock — will change)
- Do NOT modify Integrations module (low maturity)
- Do NOT modify ProductAnalytics module (low maturity)
- Do NOT modify Knowledge module (recently created)
- Do NOT modify AIKnowledge module (actively evolving)
- Do NOT change public API method signatures
- Do NOT add `#pragma warning disable` suppressions
- Do NOT modify test files for nullable annotations

## 8. Passos obrigatórios
1. Run `dotnet build 2>&1 | grep CS8632 | sort | uniq -c | sort -rn` to get warning counts per file
2. Group by module: Catalog, ChangeGovernance, IdentityAccess, AuditCompliance
3. For each high-warning file in priority order:
   - For string properties that can be null (e.g., Description, Notes): add `?` → `string?`
   - For string properties that must not be null (e.g., Name, Code): add `= string.Empty` or `= null!` in constructor
   - For navigation properties: add `?` if optional relationship
   - For required references: add guard clause in constructor
4. For DTOs and response types: ensure nullable matches the actual API behavior
5. Run `dotnet build` after each module to track progress
6. Target: < 100 CS8632 warnings total across the 4 modules
7. Run all tests for the 4 modules to verify no behavioral changes

## 9. Responsabilidades permitidas
- Add nullable annotations (`?`, `!`, `= null!`, `= string.Empty`)
- Add guard clauses in constructors for required parameters
- Annotate DTO properties to match API contracts

## 10. Responsabilidades proibidas
- Do NOT change business logic
- Do NOT change method signatures in public interfaces
- Do NOT suppress warnings with pragmas
- Do NOT fix warnings in mock/stub modules
- Do NOT refactor methods or classes

## 12. Validações obrigatórias

## 13. Riscos e cuidados
- LOW RISK: annotations don't change runtime behavior
- CAUTION: `= null!` is a null-forgiveness operator — only use for properties initialized through EF Core materialization or deserialization
- CAUTION: Changing a property from `string` to `string?` in a DTO may affect API contract serialization (null now allowed) — verify this is the desired behavior
