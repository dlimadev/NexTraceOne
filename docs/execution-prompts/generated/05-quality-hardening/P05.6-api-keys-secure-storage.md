# PROMPT — Move API keys from appsettings.json to encrypted database storage

## 1. Identification
- **Prompt ID:** P05.6
- **Phase:** 05-quality-hardening
- **Execution order:** 44
- **Type:** security
- **Area:** backend
- **Module:** identityaccess
- **Bounded context:** api-keys
- **Can run in parallel:** yes (group-E)
- **Risk level:** medium

## 2. Objective
Create an encrypted API key store in the IdentityAccess module that replaces appsettings.json as the primary storage for sensitive credentials (AI provider keys, integration webhook secrets). Keep appsettings as a development fallback.

## 3. Problem actual
Per forensic audit (`security-identity-access-report.md` and `configuration-and-parameterization-report.md`): API keys for Ollama, OpenAI, integration webhooks, and SMTP are stored in appsettings.json configuration files. In self-hosted enterprise deployments, config file secrets are a security risk — they're visible in source control, file system, and process dumps.

## 4. Resultado esperado
- New `ApiKeyEntry` entity in IdentityAccess domain with AES-256-GCM encrypted values
- `IApiKeyStore` interface for CRUD operations
- Admin-only management endpoints for API key lifecycle
- AI provider configuration reads from database store first, appsettings fallback second
- Existing functionality unchanged — just source of keys changes
- Development mode continues working with appsettings (no DB required for quick dev start)

## 5. Escopo permitido
- **Modules:**

  - `src/modules/aiknowledge/` — configuration change to read from IApiKeyStore

## 8. Passos obrigatórios
1. **Create domain entity** `ApiKeyEntry`:
   - `Id` (Guid), `KeyName` (string, unique), `EncryptedValue` (byte[]), `Provider` (string — e.g., "Ollama", "OpenAI", "SMTP")
   - `Nonce` (byte[] — for AES-GCM), `Tag` (byte[] — for AES-GCM authentication)
   - `CreatedAt` (DateTimeOffset), `CreatedBy` (Guid), `IsActive` (bool), `TenantId` (Guid)
2. **Create `IApiKeyStore` interface**:
   - `GetDecryptedKeyAsync(string keyName, CancellationToken ct) → string?`
   - `SetKeyAsync(string keyName, string plainValue, string provider, CancellationToken ct)`
   - `DeleteKeyAsync(Guid id, CancellationToken ct)`
   - `ListKeysAsync(CancellationToken ct) → List<ApiKeyMetadata>` (no decrypted values)
3. **Implement `EfApiKeyStore`**:
   - Use AES-256-GCM from `NexTraceOne.BuildingBlocks.Security` (verify existing encryption utilities)
   - Encryption key from configuration: `Security:ApiKeyEncryptionKey` (or `DATA_PROTECTION_KEY` env var)
   - Encrypt on write, decrypt on read
4. **Add DbSet<ApiKeyEntry> to IdentityDbContext**
5. **Create handlers**:
   - `ListApiKeys` — returns metadata only (name, provider, created date, active) — `admin:security:read` permission
   - `SetApiKey` — encrypts and persists — `admin:security:write` permission
   - `DeleteApiKey` — soft delete or hard delete — `admin:security:write` permission
6. **Register endpoints**:
   - `GET /api/v1/admin/api-keys` → ListApiKeys
   - `POST /api/v1/admin/api-keys` → SetApiKey
   - `DELETE /api/v1/admin/api-keys/{id}` → DeleteApiKey
7. **Update AI provider configuration**:
   - In Ollama/OpenAI provider initialization: try `IApiKeyStore.GetDecryptedKeyAsync("ollama-api-key")` first
   - If null, fall back to `IConfiguration["AIProviders:Ollama:ApiKey"]`
   - Document the fallback chain
8. **Add unit tests**:
   - Encryption roundtrip (encrypt → decrypt → original value)
   - Key not found returns null
   - List returns metadata without decrypted values
   - Fallback to appsettings works when DB store is empty
