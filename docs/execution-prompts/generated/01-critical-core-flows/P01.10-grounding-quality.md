# PROMPT — Expand AI grounding to retrieve real data from Catalog and ChangeIntelligence

## 1. Identification
- **Prompt ID:** P01.10
- **Phase:** 01-critical-core-flows
- **Execution order:** 10
- **Type:** feature-improve
- **Area:** backend
- **Module:** aiknowledge
- **Bounded context:** ai-grounding
- **Can run in parallel:** yes (with P01.2 — group-B)
- **Risk level:** low

## 2. Objective
Expand the AI grounding data retrieval to query real data from Catalog (services, ownership, dependencies) and ChangeIntelligence (recent changes, deployments) existing repositories. Currently, `DatabaseRetrievalService` only searches AI model data. This prompt adds cross-module read-only queries so the AI assistant can provide contextually rich, grounded responses about services, changes, and incidents.

## 3. Problem actual
Per forensic audit: "DB retrieval only searches AIModels, no cross-module entity lookup, no vector/semantic search. Grounding context assembly wired but data sources limited." The grounding pipeline (P9.5) is wired — `SendAssistantMessage` already calls context enrichment services. However, the actual data sources are narrow: `DatabaseRetrievalService` only queries AI-internal tables (model registry). It does not query the Service Catalog for service context, ChangeIntelligence for recent changes, or IncidentDbContext for recent incidents. The AI assistant answers questions about services and changes without access to real data.

## 4. Resultado esperado
- `DatabaseRetrievalService` queries `ServiceAssetRepository` (or Catalog DbContext) for service context when `ServiceId` is provided in the grounding request
- `DatabaseRetrievalService` queries ChangeIntelligence entities for recent changes affecting the service/environment
- `DatabaseRetrievalService` queries IncidentDbContext for recent incidents related to the service
- `DocumentRetrievalService` uses `KnowledgeDocumentRepository` if knowledge documents exist for the service or topic
- All cross-module queries are read-only — no writes to external modules
- Grounding failures are silent (logged but don't block the AI response)
- Grounding context is formatted as structured text suitable for LLM consumption

## 8. Passos obrigatórios
1. Open `DatabaseRetrievalService.cs` — understand current implementation: what it queries, what it returns, what interface it implements.
2. Add service context retrieval:
   - If `ServiceId` is provided in the grounding request, query CatalogDbContext (or ServiceAssetRepository via DI) for:
     - Service name, description, ownership (team), environment
     - Service dependencies (upstream/downstream)
     - Service metadata and classification
   - Format as structured text: `"Service: {name}, Team: {team}, Environment: {env}, Dependencies: [...]"`
3. Add recent changes retrieval:
   - Query ChangeIntelligenceDbContext for changes affecting the service within a configurable time window (default: 7 days)
   - Return: change ID, type, description, timestamp, environment, status
   - Format as structured text: `"Recent changes: [{description} on {date} in {env}]"`
   - Limit to most recent N changes (default: 10) to avoid context overflow
4. Add recent incidents retrieval:
   - Query IncidentDbContext for incidents related to the service within a configurable time window (default: 30 days)
   - Return: incident ID, title, severity, status, timestamp
   - Format as structured text: `"Recent incidents: [{title} ({severity}) on {date}]"`
   - Limit to most recent N incidents (default: 5)
5. Open `DocumentRetrievalService.cs` — add knowledge document query:
   - If service or topic is specified, query KnowledgeDocumentRepository for relevant documents
   - Return: document title, summary/excerpt
   - Format as structured text for LLM consumption
   - If KnowledgeDocumentRepository is not available or empty, return empty result silently
6. Ensure all cross-module queries are wrapped in try-catch:
   - On failure: log warning (structured logging with service ID, query type, error)
   - Return empty grounding for that data source
   - Never let a grounding failure block the AI response
7. Register cross-module DbContexts in AIKnowledge DI if not already available:
   - CatalogDbContext (read-only)
   - ChangeIntelligenceDbContext (read-only)
   - IncidentDbContext (read-only)
8. Add unit tests:
   - DatabaseRetrievalService returns service context when ServiceId provided (mock CatalogDbContext)
   - DatabaseRetrievalService returns recent changes for service (mock ChangeIntelligenceDbContext)
   - DatabaseRetrievalService returns recent incidents for service (mock IncidentDbContext)
   - Grounding failure is silent — returns empty result, does not throw
   - DocumentRetrievalService returns document excerpts when available
