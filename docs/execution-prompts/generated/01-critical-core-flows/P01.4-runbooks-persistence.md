# PROMPT — Replace hardcoded runbooks with RunbookRecord persistence and CRUD endpoints

## 1. Identification
- **Prompt ID:** P01.4
- **Phase:** 01-critical-core-flows
- **Execution order:** 6
- **Type:** feature-implement
- **Area:** backend
- **Module:** operationalintelligence
- **Bounded context:** runbooks
- **Can run in parallel:** yes (with P01.1, P01.6 — group-A)
- **Risk level:** medium

## 2. Objective
Replace the 3 hardcoded runbooks currently returned by IIncidentStore with real RunbookRecord persistence via IncidentDbContext and expose CRUD endpoints for creating, listing, and retrieving runbooks.

## 3. Problem actual
Per forensic audit: "Runbooks: 3 hardcoded in code, not via RunbookRecord." The `ListRunbooks` handler calls `store.GetRunbooks()` which returns a fixed in-memory list. The `GetRunbookDetail` handler calls `store.GetRunbookDetail()` which also returns fabricated data. The `RunbookRecord` entity already exists in `NexTraceOne.OperationalIntelligence.Domain/Incidents/Entities/RunbookRecord.cs` but is not used for persistence. IncidentDbContext may or may not have a `DbSet<RunbookRecord>` configured — verify.

## 4. Resultado esperado
- `ListRunbooks` handler reads runbooks from IncidentDbContext via repository, not from hardcoded store
- `GetRunbookDetail` handler reads from IncidentDbContext via repository, not from hardcoded store
- A `CreateRunbook` command handler persists new RunbookRecord entries
- API endpoints: `POST /api/v1/runbooks`, `GET /api/v1/runbooks`, `GET /api/v1/runbooks/{id}`
- Seed data migrated: the 3 existing hardcoded runbooks are created as seed data via IncidentSeedData or equivalent
- Zero hardcoded runbook data remains in active handler logic
