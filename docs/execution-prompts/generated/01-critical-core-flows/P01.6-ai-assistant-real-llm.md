# PROMPT — Make SendAssistantMessage invoke a real LLM instead of hardcoded responses

## 1. Identification
- **Prompt ID:** P01.6
- **Phase:** 01-critical-core-flows
- **Execution order:** 8
- **Type:** feature-implement
- **Area:** backend
- **Module:** aiknowledge
- **Bounded context:** ai-assistant
- **Can run in parallel:** yes (with P01.1, P01.4 — group-A)
- **Risk level:** high

## 2. Objective
Replace the hardcoded response logic in `SendAssistantMessage` handler with a real LLM invocation. Use the existing `IExternalAIRoutingPort` abstraction to resolve the preferred AI provider (Ollama localhost:11434 with qwen3.5:9b is configured and enabled). Pass system prompt from `AiAgentRuntimeService.BuildSystemPrompt`, pass grounding context already wired via P9.5, and handle provider errors gracefully with fallback to deterministic response.

## 3. Problem actual
Per forensic audit: "SendAssistantMessage returns hardcoded responses without LLM integration. Ollama localhost:11434, qwen3.5:9b configured and enabled. IExternalAIRoutingPort exists as abstraction." The handler currently constructs a response with a fixed string (or deterministic fallback) rather than calling any AI provider. The infrastructure to call an LLM exists — `IChatCompletionProvider` (Ollama/OpenAI implementations in `Infrastructure/Runtime/Providers/`), `IExternalAIRoutingPort` for provider resolution, and `AiAgentRuntimeService` for system prompt building — but `SendAssistantMessage` does not use them.

## 4. Resultado esperado
- `SendAssistantMessage` handler calls `IChatCompletionProvider` with the user's message, system prompt, and grounding context
- Provider is resolved via `IExternalAIRoutingPort` based on user preferences, policies, and available models
- System prompt is built via `AiAgentRuntimeService.BuildSystemPrompt()` incorporating persona, context scope, and grounding data
- Grounding context (already wired in P9.5) is included in the prompt/context
- On provider failure: graceful degradation to deterministic fallback with clear error indication in response metadata
- Response includes: AI-generated content, model used, provider ID, token usage estimates, grounding sources used
- Existing governance checks (policy validation, budget checks) remain intact

## 8. Passos obrigatórios
1. Open `SendAssistantMessage.cs` handler — locate the section that builds the response content. Identify where the hardcoded/deterministic response is generated.
2. Verify the handler's constructor dependencies — ensure it injects (or can inject):
   - `IChatCompletionProvider` (or a factory/resolver that provides it)
   - `IExternalAIRoutingPort` (for resolving provider/model selection)
   - `IAiAgentRuntimeService` (for building system prompt)
3. In the handler's `Handle` method, after existing governance checks (policy validation, budget, etc.):
   a. Resolve the preferred AI provider/model using `IExternalAIRoutingPort` — pass the user's `PreferredModelId`, persona, and context scope
   b. Build the system prompt via `AiAgentRuntimeService.BuildSystemPrompt()` — pass persona, context scope, and grounding context
   c. Construct the chat completion request with: system prompt, conversation history (if existing conversation), user message, grounding context
   d. Call `IChatCompletionProvider.CompleteAsync()` (or equivalent method) with the constructed request
   e. Extract response content, token usage, model info from the provider response
4. Handle provider errors gracefully:
   - If provider is unreachable: fall back to `DegradedProviderId` / `DegradedModelName` constants already in the handler
   - Log the error with structured logging (provider ID, error type, conversation ID)
   - Return a response with `IsDegraded = true` (or equivalent metadata flag)
   - Include a user-friendly error indication in the response metadata
5. Ensure the response record includes:
   - `Content`: AI-generated text (or fallback text)
   - `ModelUsed`: actual model name from provider response
   - `ProviderId`: resolved provider ID
   - `TokensUsed`: token count from provider (estimate if not available)
   - `GroundingSources`: list of grounding sources that contributed context
   - `IsDegraded`: boolean indicating fallback mode
6. Preserve all existing governance logic: policy validation, conversation ownership, audit trail
7. Add unit tests mocking `IChatCompletionProvider`:
   - Successful LLM call returns AI-generated content
   - Provider failure triggers graceful degradation
   - Routing resolves correct provider based on preferences
   - System prompt includes persona and grounding context
   - Token usage is captured in response

- HIGH RISK: LLM calls are latent and may timeout — ensure CancellationToken is propagated and reasonable timeout is configured
- CAUTION: IChatCompletionProvider may have different method signatures for Ollama vs OpenAI — verify the abstraction is provider-agnostic

- CAUTION: Grounding context may be large — ensure it doesn't exceed provider's context window. Truncate if necessary.
- CAUTION: Conversation history should be bounded (last N messages) to avoid context overflow
