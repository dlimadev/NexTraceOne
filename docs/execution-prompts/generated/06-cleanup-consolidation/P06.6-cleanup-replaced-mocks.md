## 2. Objective
Remove dead mock/fabricated code from Governance application handlers that have been confirmed replaced by real implementations in P03.1 (Teams/Domains), P03.2 (remaining governance handlers), and P03.3 (FinOps handlers). After P03.4 completes the `IsSimulated` property removal, the old fabrication code, commented-out blocks, resolved TODO markers, and orphaned mock-data helper methods become dead code that must be cleaned up to maintain codebase hygiene.

## 3. Problem actual
After Phase 03 replaces mock handlers with real cross-module data, the old mock/fabricated code may persist in several forms:

1. **Commented-out blocks:** Developers may comment out old fabrication logic rather than deleting it during P03.1/P03.2/P03.3, creating code noise.
2. **Resolved TODO markers:** Comments like `// TODO: replace with real data` that are now resolved but not cleaned up.
3. **Orphaned fabrication helpers:** Private methods like `GenerateFakeTeams()`, `BuildMockDomainSummary()`, `CreateSampleFinOpsData()` that were used by fabrication logic but are no longer called after the handler was rewritten.
4. **Mock data constants:** Static arrays or dictionaries containing hardcoded team names, service counts, compliance scores, financial figures used for fabrication.
5. **Dead using statements:** `using` directives for namespaces that were only needed by fabrication code.

These residues make the codebase harder to maintain, increase cognitive load during code review, and may confuse future contributors about which code is active.

Evidence path:
- `src/modules/governance/NexTraceOne.Governance.Application/Features/` — 60 feature handler directories, up to 74 handlers modified across P03.1/P03.2/P03.3/P03.4

## 8. Passos obrigatórios
1. **Review P03.4 summary** — Read the P03.4 execution deliverable that lists which handlers now return real data and which (if any) remain simulated. This is the authoritative guide for which handlers can be cleaned.
2. **For each confirmed-real handler, systematically check for:**
   - **Commented-out mock code:** Search for patterns like `// var fake`, `// return new List`, `/* fabricated */`, multi-line comment blocks containing mock logic
   - **Resolved TODO markers:** Search for `// TODO: replace with real`, `// TODO: remove mock`, `// HACK:`, `// TEMPORARY:` comments that reference mock data replacement
   - **Orphaned fabrication methods:** Private methods not called by any active code path — search for `Generate`, `Fake`, `Mock`, `Sample`, `Fabricate`, `Simulated`, `Hardcoded` in method names
   - **Dead mock data constants:** Static fields/properties containing hardcoded data arrays, lists, or dictionaries that are no longer referenced
   - **Dead using statements:** Using directives for namespaces only needed by removed fabrication code
3. **For each handler still simulated (if any remain):** Leave completely untouched — do not clean, do not modify.
4. **Remove dead code** — For each confirmed-real handler:
   - Delete commented-out mock code blocks
   - Delete resolved TODO/HACK/TEMPORARY comments
   - Delete orphaned private fabrication methods
   - Delete unreferenced mock data constants
   - Delete unused using statements
   - Preserve all active code: production logic, logging, validation, error handling, CancellationToken propagation
5. **Verify compilation** — Run:
   ```
   dotnet build src/modules/governance/NexTraceOne.Governance.Application/
   ```
6. **Verify tests** — Run:
   ```
   dotnet test tests/modules/governance/
   ```
7. **Final verification scan** — Confirm cleanup is complete:
   ```
   grep -rn "TODO.*mock\|TODO.*fake\|TODO.*simulated\|TODO.*replace.*real\|HACK\|TEMPORARY" src/modules/governance/NexTraceOne.Governance.Application/Features/ --include="*.cs"
   grep -rn "Fabricat\|FakeData\|MockData\|SampleData\|HardcodedData" src/modules/governance/NexTraceOne.Governance.Application/Features/ --include="*.cs"
   ```

8. **Document results** — For each cleaned handler, record: file path, lines removed, type of dead code removed (commented block / TODO / orphaned method / constant / using).

## 13. Riscos e cuidados
- **MEDIUM RISK** — modifying production handler files, though only removing dead code
- **False positive dead code:** Before deleting any method or constant, verify it is truly unreferenced. Use IDE "Find All References" or `grep -rn "MethodName"` to confirm zero callers. A method that appears orphaned might be called via reflection, MediatR pipeline, or base class override.
- **P03.4 dependency is critical:** This prompt MUST NOT execute before P03.4 confirms which handlers are real. Cleaning a handler that P03.4 hasn't verified could remove code that is still active.
- **Test coupling:** Some test files may directly call fabrication helper methods for test setup. If so, update tests to use proper test fixtures or mock interfaces instead.
- **Partial cleanup ambiguity:** If P03.1/P03.2/P03.3 left some handlers partially converted (real data for some fields, mock for others), do NOT remove the mock code for the still-mock fields. Only remove code for fields confirmed real.
- **Using statement removal:** Be careful removing using statements — verify the namespace is truly unused. IDE tooling or `dotnet build` will catch errors, but be methodical.
