# PROMPT — Create a single authoritative ROADMAP reflecting audit findings and execution prompts sequence

## 1. Identification
- **Prompt ID:** P06.5
- **Phase:** 06-cleanup-consolidation
- **Execution order:** 49
- **Type:** cleanup
- **Area:** documentation
- **Module:** cross-cutting
- **Bounded context:** roadmap
- **Can run in parallel:** yes (group-F)
- **Risk level:** low

## 2. Objective
Create a single authoritative `docs/ROADMAP-AUTHORITATIVE.md` that reflects the forensic audit findings, execution prompts sequence, and product pillar status as the definitive source of truth for project direction. Add deprecation notices to any older roadmap documents. The authoritative roadmap must be honest — clearly marking what is real, what is mock/stub, and what is planned but not yet started.

## 3. Problem actual
Multiple roadmap and status documents exist across the repository with varying levels of currency and accuracy:

- `docs/ROADMAP.md` (if exists at root) may describe an outdated view of the project
- `docs/archive/old-roadmaps/` contains one or more superseded roadmap versions
- The execution prompts sequence (`docs/execution-prompts/EXECUTION-PROMPTS-SEQUENCE.md`) now serves as the de facto detailed task plan but lacks a high-level strategic view
- `docs/PLATFORM-CAPABILITIES.md` describes capabilities but may not reflect actual implementation state
- `docs/PRODUCT-VISION.md` describes the aspirational vision but not the current-state-to-vision bridge

After all audit phases and execution prompts are defined, a single authoritative roadmap should exist that connects current state → execution plan → product vision, so that any contributor or stakeholder can understand where the project stands and where it is headed.

## 4. Resultado esperado
- A single `docs/ROADMAP-AUTHORITATIVE.md` that serves as the authoritative project roadmap
- Clear separation between: what's done (completed phases), what's in progress (current execution prompts), what's upcoming (remaining phases), and what's aspirational (future product vision)
- Product Pillars Status section drawn from the capability-gap-matrix and forensic audit
- Honest marking of mock/stub/planned items — no false claims of readiness
- Deprecation notices on older roadmap files directing readers to the authoritative version
- The authoritative roadmap references execution prompts for detailed task breakdowns (does not duplicate them)

## 5. Escopo permitido
- **Directories:** `docs/` (top-level markdown files and `docs/archive/old-roadmaps/`)
- **Files:** New `docs/ROADMAP-AUTHORITATIVE.md`, old ROADMAP.md (deprecation notice), DOCUMENTATION-INDEX.md (reference update)
- **Reference sources:** `docs/execution-prompts/EXECUTION-PROMPTS-SEQUENCE.md`, `docs/audit-forensic-2026-03/`, `docs/PRODUCT-VISION.md`, `docs/PLATFORM-CAPABILITIES.md`, `docs/current-state/` (if created by P00.2)
- **Artifact types:** Markdown documentation

## 6. Escopo proibido
- Do NOT delete old roadmap files — add deprecation notices only
- Do NOT modify source code under `src/` or `tests/`
- Do NOT modify `docs/PRODUCT-VISION.md` — product vision is the aspirational north star and must not be conflated with the roadmap
- Do NOT modify `docs/audit-forensic-2026-03/` — forensic audit is source evidence
- Do NOT modify execution prompt files — the roadmap references them, not replaces them
- Do NOT make promises about features that have no execution prompt or clear plan
- Do NOT fabricate completion dates or timelines — the roadmap describes sequence and dependencies, not calendar dates
- Do NOT modify infrastructure or configuration files

## 7. Ficheiros principais candidatos a alteração
1. `docs/ROADMAP-AUTHORITATIVE.md` (new — the canonical authoritative roadmap)
2. `docs/ROADMAP.md` (add deprecation header, if file exists at root)
3. `docs/DOCUMENTATION-INDEX.md` (update to reference the authoritative roadmap as primary)
4. Any other roadmap-like documents found during inventory (add deprecation headers)

## 8. Passos obrigatórios
1. **Inventory existing roadmaps** — Search for all roadmap-related documents:
   ```
   find docs/ -iname "*roadmap*" -o -iname "*road-map*"
   grep -rln "roadmap\|road map" docs/ --include="*.md" | head -20
   ```
   Also check `docs/archive/old-roadmaps/` for context on superseded roadmaps.
2. **Gather source data** — Read and extract key information from:
   - `docs/execution-prompts/EXECUTION-PROMPTS-SEQUENCE.md` — phases, gate structure, execution order
   - `docs/execution-prompts/GATES-AND-DEPENDENCIES.md` — dependency graph, gate criteria
   - `docs/audit-forensic-2026-03/final-project-state-assessment.md` — current state assessment
   - `docs/PRODUCT-VISION.md` — aspirational vision and product pillars
   - `docs/PLATFORM-CAPABILITIES.md` — capability inventory
   - `docs/current-state/` (if exists) — per-module current state
3. **Create `docs/ROADMAP-AUTHORITATIVE.md`** with the following structure:
   - **Title and metadata:** Document purpose, creation date, source references, relationship to PRODUCT-VISION.md
   - **Current State Summary:** Brief overview linking to `docs/CURRENT-STATE-SUMMARY.md` (P06.2) and `docs/current-state/` (P00.2) for details
   - **Product Pillars Status:** Table with one row per product pillar (from Section 3 of copilot instructions): pillar name, current maturity level (None / Stub / Partial / Functional / Complete), key gaps, execution prompts addressing it
   - **Completed Phases:** List of execution prompt phases that have been completed, with summary of what each delivered
   - **In-Progress Phases:** Current execution prompt phases being executed, with summary of scope
   - **Upcoming Phases:** Remaining execution prompt phases, with summary of what each will deliver
   - **Execution Prompts Reference:** Link to `docs/execution-prompts/EXECUTION-PROMPTS-SEQUENCE.md` for the detailed 50-prompt execution plan
   - **Gate Structure:** Brief overview of quality gates between phases, linking to GATES-AND-DEPENDENCIES.md for details
   - **Aspirational Vision:** Brief link to PRODUCT-VISION.md noting the long-term north star
   - **Honesty section:** Explicit callout: "Items marked MOCK, STUB, or PLANNED are not yet functional. See individual execution prompts for remediation scope."
4. **Mark mock/stub/planned honestly** — For every capability or module referenced in the roadmap:
   - If fully implemented and verified: mark ✅ REAL
   - If partially implemented: mark 🔶 PARTIAL with gap note
   - If mock/stub: mark ⚠️ MOCK/STUB
   - If planned but not started: mark 📋 PLANNED
5. **Add deprecation notices** — For any older roadmap documents found in step 1:
   ```
   > ⚠️ **DEPRECATED** — This roadmap has been superseded. See [`docs/ROADMAP-AUTHORITATIVE.md`](./ROADMAP-AUTHORITATIVE.md) for the current authoritative project roadmap.
   ```
6. **Update documentation index** — Ensure `docs/DOCUMENTATION-INDEX.md` lists `ROADMAP-AUTHORITATIVE.md` as the primary roadmap reference and marks older versions as deprecated.
7. **Cross-reference verification** — Check that all links in the new roadmap resolve to existing files.

## 13. Riscos e cuidados
- **LOW RISK** — documentation-only changes, no code impact
- **Accuracy risk:** The roadmap must not overstate project maturity. Every claim must be traceable to audit evidence or execution prompt scope. When in doubt, mark as PLANNED or PARTIAL.
- **Vision vs. roadmap confusion:** The roadmap describes the path from current state to vision; PRODUCT-VISION.md describes the destination. Keep them clearly separated.
- **Execution prompts evolution:** The execution prompts sequence may evolve as phases execute. The roadmap should reference the sequence document rather than hardcoding prompt details, so it stays current.
- **Stakeholder expectations:** An honest roadmap may reveal uncomfortable gaps. This is intentional and aligned with the NexTraceOne Source of Truth principle — transparency over comfort.
