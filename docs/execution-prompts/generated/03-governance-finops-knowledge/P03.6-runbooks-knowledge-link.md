# PROMPT — Link RunbookRecord to Knowledge Hub via KnowledgeRelation for AI Grounding

## 1. Identification
- **Prompt ID:** P03.6
- **Phase:** 03-governance-finops-knowledge
- **Execution order:** 31
- **Type:** feature-implement
- **Area:** Knowledge
- **Module:** knowledge
- **Bounded context:** knowledge-runbooks
- **Can run in parallel:** no (depends on P03.5 Knowledge Hub persistence being verified)
- **Risk level:** low

## 2. Objective
Create KnowledgeRelation entries that link RunbookRecord entities (from OperationalIntelligence/Incidents) to their associated services in the Knowledge Hub, enabling runbooks to be discoverable via Knowledge search and AI grounding. When a RunbookRecord is created or updated, a corresponding KnowledgeRelation should be created linking it to its service context. Optionally, create OperationalNote summaries for runbooks to enrich the Knowledge Hub searchable corpus.

## 3. Problem actual
Runbooks (created by P01.4 as RunbookRecord in IncidentDbContext at `src/modules/operationalintelligence/NexTraceOne.OperationalIntelligence.Domain/Incidents/Entities/RunbookRecord.cs`) and the Knowledge Hub (verified by P03.5) are not connected. The KnowledgeRelation entity at `src/modules/knowledge/NexTraceOne.Knowledge.Domain/Entities/KnowledgeRelation.cs` supports linking to various entity types (Service, Contract, Change, Incident) but no integration exists to automatically create these links when runbooks are managed.

This means:
- Runbooks are invisible to Knowledge Hub search
- AI grounding cannot find runbooks when investigating incidents for a specific service
- The "Source of Truth & Operational Knowledge" product pillar is incomplete — operational runbooks exist but are siloed in the Incidents subdomain
- Engineers searching for "how to handle database timeout for service X" won't find the runbook even if one exists

Evidence:
- `src/modules/operationalintelligence/NexTraceOne.OperationalIntelligence.Domain/Incidents/Entities/RunbookRecord.cs` — RunbookRecord entity with service association
- `src/modules/knowledge/NexTraceOne.Knowledge.Domain/Entities/KnowledgeRelation.cs` — supports cross-entity linking
- `src/modules/knowledge/NexTraceOne.Knowledge.Application/Features/CreateKnowledgeRelation/CreateKnowledgeRelation.cs` — handler exists
- `src/modules/knowledge/NexTraceOne.Knowledge.Application/Features/GetKnowledgeByRelationTarget/GetKnowledgeByRelationTarget.cs` — query handler exists
- `src/modules/operationalintelligence/NexTraceOne.OperationalIntelligence.Infrastructure/Incidents/Persistence/IncidentDbContext.cs` — RunbookRecord persistence

## 4. Resultado esperado
- When a RunbookRecord is created or updated, a KnowledgeRelation is created linking it to the associated service ID in the Knowledge Hub.
- GetKnowledgeByRelationTarget can find runbooks when querying by service ID.
- Optionally, an OperationalNote summary is created for each runbook, making runbook content searchable via FTS.
- Runbooks become discoverable through Knowledge Hub search and AI grounding workflows.
- The link is maintained: if a runbook is updated, the KnowledgeRelation remains valid.
- Unit tests verify the complete roundtrip: create runbook → create relation → query by service → find runbook reference.

## 8. Passos obrigatórios
1. **Analyze KnowledgeRelation entity** — Review `KnowledgeRelation.cs` to understand supported SourceType/TargetType values, required fields (SourceEntityId, TargetEntityId, SourceType, TargetType, etc.), and whether a "Runbook" relation type exists or needs to be added as a value.
2. **Create runbook-to-knowledge linking service or handler** — Implement a service or handler that:
   - Accepts a RunbookRecord's ID, associated service ID, title, and optionally content summary
   - Creates a KnowledgeRelation with: SourceType = "Runbook" (or equivalent enum value), SourceEntityId = runbook ID, TargetType = "Service", TargetEntityId = service ID
   - Persists via IKnowledgeRelationRepository.AddAsync()
   - This can be a new MediatR command (e.g., `LinkRunbookToKnowledgeCommand`) or an application service
3. **Integrate with RunbookRecord creation flow** — In the OperationalIntelligence Incidents application layer, after a RunbookRecord is successfully created/updated:
   - Option A (preferred): Publish a domain event (e.g., `RunbookCreatedEvent`) that the Knowledge module handles via MediatR notification
   - Option B: Call the Knowledge module's linking service directly from the runbook creation handler
   - Option C: Use an integration event via the event bus if available
4. **Verify GetKnowledgeByRelationTarget query** — Ensure `GetKnowledgeByRelationTarget.cs` can find runbook relations when querying by service ID. If it filters by specific TargetType values, add "Runbook" to supported types.
5. **Optionally create OperationalNote for runbook** — When a runbook is linked, optionally create an OperationalNote in the Knowledge Hub containing the runbook title, description, and key steps. This makes runbook content searchable via FTS (from P03.5).
6. **Add unit tests** — Test the complete roundtrip:
   - Test: Create a mock RunbookRecord → invoke linking service → verify KnowledgeRelation created with correct SourceType/TargetType/IDs
   - Test: Given a KnowledgeRelation exists for a runbook → query GetKnowledgeByRelationTarget with service ID → verify runbook relation returned
   - Test: Verify linking service handles duplicate creation gracefully (idempotent or update)
7. **Verify compilation** — Run `dotnet build` on both knowledge and operationalintelligence modules.

## 13. Riscos e cuidados
- **Cross-module coupling:** The biggest risk is introducing tight coupling between OperationalIntelligence and Knowledge modules. Prefer domain events or published interfaces over direct service references.
- **Event ordering:** If using domain events, ensure the KnowledgeRelation is created after the RunbookRecord is committed to the database. Use eventual consistency patterns if needed.
- **KnowledgeRelation type enum:** If KnowledgeRelation uses a closed enum for SourceType/TargetType, adding "Runbook" requires modifying the domain entity — which is prohibited. In that case, use an existing compatible type (e.g., "Document") or propose adding the type via a domain enum extension that doesn't change entity structure.
- **Idempotency:** If a runbook is updated multiple times, the linking logic should not create duplicate KnowledgeRelations. Implement upsert or check-before-create logic.
- **AI grounding integration:** This prompt enables AI grounding for runbooks but does not implement the AI query flow itself. That's a separate concern in the AI module.
