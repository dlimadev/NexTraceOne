## 2. Objective
Implement the `IExternalAiModule` cross-module interface defined in `NexTraceOne.AIKnowledge.Contracts/ExternalAI/ServiceInterfaces/IExternalAiModule.cs`. Currently the interface is empty (no methods defined). This prompt requires first defining the interface methods based on the `ExternalAiDbContext` entities and cross-module consumption needs, then implementing them. The interface enables other modules — particularly Governance (AI provider governance and cost tracking), Configuration (provider health monitoring), and Audit (external AI usage audit) — to query external AI provider and policy data without referencing the ExternalAI DbContext directly.

## 3. Problem actual
The `IExternalAiModule` interface exists but is empty — no methods are defined:

```csharp
public interface IExternalAiModule
{
    // Empty — planned but not designed
}
```

`ExternalAiDbContext` has entities: `ExternalAiProvider` (registered external AI providers), `ExternalAiPolicy` (governance policies for external AI usage), `ExternalAiConsultation` (queries sent to external providers), and `KnowledgeCapture` (organizational knowledge captured from AI interactions). After P01.9 (ExternalAI handler implementation), providers will have real health data and model registrations. Other modules need to query this data for:
- Provider availability and health status
- Governance compliance checks
- Route selection for AI requests based on policy
- External AI usage audit reports

## 4. Resultado esperado
- `IExternalAiModule` interface defined with the following methods:
  - `GetAvailableProvidersAsync(CancellationToken ct)` → returns list of registered AI providers with their status
  - `GetProviderHealthAsync(Guid providerId, CancellationToken ct)` → returns health status of a specific provider
  - `RouteRequestAsync(string capability, string? preferredProvider, CancellationToken ct)` → returns the best provider for a given capability based on policy and availability
- `ExternalAiModule` class implementing all methods, reading from `ExternalAiDbContext`
- DTOs defined in the Contracts project for the return types
- Registration in DI

## 8. Passos obrigatórios
1. **Inspect ExternalAiDbContext entities** to understand available data:
   - `ExternalAiProvider`: provider id, name, type (Ollama/OpenAI/etc), endpoint, status, capabilities, last health check
   - `ExternalAiPolicy`: policy rules for AI provider usage (which providers, which capabilities, rate limits)
   - `ExternalAiConsultation`: queries sent to external providers (for usage tracking)
   - `KnowledgeCapture`: captured knowledge from AI interactions
2. **Define DTOs in Contracts project:**
   - `ProviderSummaryDto`: Id, Name, ProviderType, Status, Capabilities, LastHealthCheck
   - `ProviderHealthDto`: ProviderId, Status (Healthy/Unhealthy/Degraded), Latency, LastChecked, ErrorMessage
   - `RoutingDecisionDto`: SelectedProviderId, ProviderName, Reason, FallbackProviderId
3. **Define interface methods** in `IExternalAiModule`:
   - `Task<IReadOnlyList<ProviderSummaryDto>> GetAvailableProvidersAsync(CancellationToken ct = default);`
   - `Task<ProviderHealthDto?> GetProviderHealthAsync(Guid providerId, CancellationToken ct = default);`
   - `Task<RoutingDecisionDto?> RouteRequestAsync(string capability, string? preferredProvider = null, CancellationToken ct = default);`
4. **Create `ExternalAiModule` class:**
   - Inject `ExternalAiDbContext`
   - Implement `GetAvailableProvidersAsync`: query all active providers, map to DTOs
   - Implement `GetProviderHealthAsync`: query provider by ID, return health status from last health check data
   - Implement `RouteRequestAsync`: select best provider based on capability support, policy rules, and health status
   - Use `AsNoTracking()`, propagate `CancellationToken`
5. **Register in DI:** `services.AddScoped<IExternalAiModule, ExternalAiModule>();`
6. **Verify platform registration**
7. **Add unit tests** (≥6)

## 13. Riscos e cuidados
- **MEDIUM RISK:** Defining new interface methods is a design decision — keep the interface minimal. The routing logic should be simple for now (prefer healthy provider with matching capability)
- **CRITICAL:** Never expose provider API keys, endpoints with credentials, or secret configuration through DTOs. ProviderSummaryDto should have provider type and status, NOT connection details
- **CAUTION:** The routing decision logic should be deterministic and testable — avoid complex scoring algorithms in the initial implementation
- **CAUTION:** If ExternalAiDbContext has no providers configured, all methods should return null/empty gracefully
- **CAUTION:** Provider health status may be stale — include `LastChecked` timestamp so consumers can assess freshness
