# PROMPT — Implement IContractsModule cross-module interface with real data from ContractsDbContext

## 1. Identification
- **Prompt ID:** P02.2
- **Phase:** 02-structural-foundations
- **Execution order:** 21
- **Type:** feature-implement
- **Area:** backend
- **Module:** catalog
- **Bounded context:** contracts-interface
- **Can run in parallel:** yes (with P02.3 — group-D)
- **Risk level:** medium

## 2. Objective
Implement the `IContractsModule` cross-module interface defined in `NexTraceOne.Catalog.Contracts/Contracts/ServiceInterfaces/IContractsModule.cs`. The implementation must read real data from `ContractsDbContext` (which already has migrations and domain entities) and be registered in DI in both `ApiHost/Program.cs` and `BackgroundWorkers/Program.cs`. This interface is the primary mechanism through which other bounded contexts (ChangeGovernance, OperationalIntelligence, AIKnowledge, DeveloperPortal) query contract data without directly referencing the Contracts DbContext.

## 3. Problem actual
The `IContractsModule` interface is defined with four methods but has zero implementation:

```csharp
public interface IContractsModule
{
    Task<ChangeLevel?> GetLatestChangeLevelAsync(Guid apiAssetId, CancellationToken ct = default);
    Task<bool> HasContractVersionAsync(Guid apiAssetId, CancellationToken ct = default);
    Task<decimal?> GetLatestOverallScoreAsync(Guid apiAssetId, CancellationToken ct = default);
    Task<bool> RequiresWorkflowApprovalAsync(Guid apiAssetId, CancellationToken ct = default);
}
```

The `ContractsDbContext` exists at `src/modules/catalog/NexTraceOne.Catalog.Infrastructure/Contracts/Persistence/ContractsDbContext.cs` with working migrations and entity configurations. However, no class implements `IContractsModule`, so any module that depends on contract data must either hardcode stubs or bypass the bounded context boundary by accessing the DbContext directly (which violates the architecture rule).

This blocks:
- P03.1 (contract governance workflows)
- P01.10 (grounding quality — AI needs contract context)
- At least 7 Developer Portal stubs that need contract data to display real information
- Change Intelligence blast radius analysis (needs to know which services have contracts)

## 4. Resultado esperado
- A `ContractsModule` class implementing `IContractsModule` in the Catalog Infrastructure project
- All four methods reading real data from `ContractsDbContext` entities
- `GetLatestChangeLevelAsync` queries the latest contract version's change level for a given API asset
- `HasContractVersionAsync` checks if at least one published contract version exists
- `GetLatestOverallScoreAsync` returns the latest quality/compliance score
- `RequiresWorkflowApprovalAsync` checks if the contract's publication policy requires approval
- Registration in DI via the Catalog module's `DependencyInjection.cs` (as a scoped service)
- Verification that `ApiHost/Program.cs` and `BackgroundWorkers/Program.cs` call the Contracts module registration

## 13. Riscos e cuidados
- **MEDIUM RISK:** The ContractsDbContext entity model may have strongly-typed IDs — ensure the Guid parameter in the interface methods maps correctly to the entity's key type
- **CAUTION:** The `ChangeLevel` enum referenced in the interface must exist in the Contracts project — verify the enum definition and ensure it matches what the DbContext uses
- **CAUTION:** The "latest version" query must order by version number or creation date — verify which field represents ordering
- **CAUTION:** Tenant isolation is handled by NexTraceDbContextBase's RLS — do not add manual tenant filtering on top
- **CAUTION:** This is a read-only interface — do not accidentally trigger SaveChanges or modify tracked entities
